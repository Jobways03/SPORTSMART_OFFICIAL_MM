# Delhivery — Flow & Full Test Guide

_Last updated: 2026-06-03. UI-only test plan covering every Delhivery capability wired into the app._

## Apps & ports
| App | Role | Port |
|---|---|---|
| Storefront | Customer | 4005 |
| Super Admin | Platform admin | 4000 |
| D2C Seller — portal / admin | Seller | 4003 / 4001 |
| Retail Seller — portal / admin | Retailer | 4009 / 4008 |
| Franchise — portal / admin | Franchise | 4004 / 4002 |

Portal = the seller/franchise logs in to run their own shop. Admin = platform staff.

## How it works (one-paragraph)
Every sub-order is created with `deliveryMethod = DELHIVERY` by default. When the seller/franchise marks it **PACKED**, the system **auto-books** Delhivery (sending the product's real weight + dimensions), attaches the **AWB**, and the order goes **SHIPPED** — no manual shipping. The seller/franchise then clicks **Request pickup** to have Delhivery collect it. Tracking flows in via a **webhook** (admins can also **Refresh tracking**). Labels download per order. Cancelling is **Delhivery-first** (voids the AWB, then cancels the order + refunds); an already-picked-up parcel is handled by **Force RTO**.

## Prerequisite (one-time)
The test seller/franchise must have a **registered Delhivery pickup location**: their admin (or Settings) → **Logistics partners** → register with Delhivery. Otherwise "Request pickup" returns *"register your pickup location first."*

> Testability note: items marked **[live]** need a deployed/public URL or a real Delhivery event and **cannot** be fully exercised on localhost (Delhivery can't reach `localhost`). Everything else works against Delhivery staging from local.

## TEST 1 — Serviceability (storefront 4005)
1. Open any product → enter a serviceable pincode (e.g. `500056`) → **Check delivery**.
   - ✅ Shows deliverable / a delivery option.
2. Enter a junk/remote pincode (e.g. `190001` remote, or `000000`).
   - ✅ Shows **"Not deliverable here"** (only an explicit Delhivery `serviceable:false` blocks; a carrier error fails-open so it never wrongly blocks).

## TEST 2 — Place order → auto-book → SHIPPED (core lifecycle)
1. **Storefront (4005):** place a COD order to a serviceable address.
2. **Super Admin (4000):** Orders → open it → **Verify**.
3. **Seller portal (4003)** _(or Retail 4009 / Franchise 4004)_: open the order → **Accept** → **Mark as Packed**.
   - ✅ Order flips to **SHIPPED**, an **AWB** appears, the badge reads **"Delhivery."**
   - ✅ No manual "Mark as Shipped" button for Delhivery (auto-ships).

## TEST 3 — Weight & dimensions accuracy
1. As a seller, set a product's **Weight** + **Dimensions** (e.g. 20 kg, 40×30×14.9) and save.
2. Place + pack an order for it (Test 2).
3. On Delhivery (one.delhivery.com → the AWB), check **Package Details / Weight**.
   - ✅ Shows the **real** values (e.g. ~40×30×14.9 / 20 kg), **not** the old `10×10×10 / 500 g` fallback.

## TEST 4 — Self-service pickup (portals + tenancy)
1. In the **seller portal (4003)**: open a packed Delhivery order → **🚚 Request Delhivery pickup**.
   - ✅ Toast: *"Pickup requested with Delhivery for `<date>` (~N parcels)…"*.
   - ✅ If no pickup location registered → message tells you to register first (no silent fail).
2. Repeat from **Retail portal (4009)** and **Franchise portal (4004)**.
   - ✅ Each works for its own order.
3. **Tenancy:** a seller can only pickup **their own** orders (the API rejects another seller's sub-order with 403). Admins (4000/4001/4008/4002) can also Request pickup from the order's Shipping panel.

## TEST 5 — Shipping label (self-service in portals + admins)
The seller/franchise packs the box, so the label is downloadable **in their own portal** — they no longer depend on an admin. It's available everywhere for parity.

**Portals (4003 / 4009 / 4004):**
1. Open a packed/shipped Delhivery order → click **"🏷️ Download shipping label"** (sits next to "Request Delhivery pickup").
   - ✅ Opens the real Delhivery label PDF in a new tab.
   - ✅ If Delhivery hasn't generated it yet → friendly **"Label not generated by Delhivery yet — available once the shipment is manifested / picked up"** (no broken link / no 404).
   - ✅ **Tenancy:** the seller/franchise can only fetch **their own** order's label (another tenant's sub-order → 403).

**Admins (4000 / 4001 / 4008 / 4002):**
2. Open a shipped Delhivery order → **Shipping** panel.
   - ✅ The label auto-fetches; a **"⬇ Download shipping label"** button appears when ready (else the same "not generated yet" + **Retry**).
3. Click **Download shipping label** → opens the real Delhivery label PDF.

## TEST 6 — Tracking
- **Refresh tracking** (admin Shipping panel → Carrier actions): click it on a shipped order.
  - ✅ Pulls the latest scan from Delhivery and updates the status/last-event.
- **Open courier page** (admin Shipping panel / customer order): click the tracking link.
  - ✅ Opens `https://www.delhivery.com/track/package/<AWB>` (correct, not a 404).
- **[live] Webhook:** once Delhivery is configured to POST to your public webhook URL, a real scan updates the order automatically. On localhost, simulate with a curl to `POST /shipping/webhooks/delhivery`.

## TEST 7 — Cancel (Delhivery-first, both directions)
Both admin buttons — **"Cancel order + shipment"** (Shipping panel) and **"Cancel sub-order"** (order page) — run the same flow.
1. On a Delhivery order **not yet picked up** → click cancel → enter a reason.
   - ✅ Delhivery cancels first → order goes **CANCELLED** (admin + customer), prepaid customer refunded, AWB voided.
2. On a Delhivery order **already picked up / in transit** → click cancel.
   - ✅ **Blocked** with *"Delhivery couldn't cancel — likely already picked up. Use Force RTO…"* — the order stays put.
3. Reverse direction: cancelling the sub-order also cancels the Delhivery shipment (same handler).

## TEST 8 — Force RTO (admins)
1. Shipping panel → Carrier actions → **Force RTO** → enter a reason (≥10 chars).
   - ✅ Returns the parcel to origin AND cancels + refunds the order + restores stock (no longer leaves it stuck SHIPPED).

## TEST 9 — NDR re-attempt (admins)
- Shipping panel → Carrier actions → **Re-attempt delivery** (meaningful only after a real failed-delivery / NDR event).
  - ✅ Sends a re-attempt request to Delhivery. **[live]** — needs an actual NDR to be meaningful.

## TEST 10 — Customer-facing visibility (storefront 4005)
1. **Shipped order:** customer order page shows **Shipped** + a working **tracking** link; "My Orders" list shows the same.
2. **Cancelled order:** after Test 7, the customer order **detail** shows the red **"Cancelled"** state (tracker shows ✗ Cancelled, no "Cancel Order" button) and the **list** shows a red **Cancelled** badge — the two agree.
3. The courier badge reads **"Delhivery"** (not "Preparing").

## TEST 11 — Franchise is Delhivery-only
1. **Franchise portal (4004):** open an order.
   - ✅ Delivery method shows **"Delhivery"**; there is **no** self-delivery picker and **no** self-delivery status buttons.
   - ✅ The flow is Accept → Mark Packed → (auto-ship) → **Request pickup** — identical to seller/retail.

## TEST 12 — Admin order-status consistency
1. **Super Admin (4000) Orders list + detail:** a fully-cancelled order shows **Cancelled** in both the order-status column and the fulfillment column (no "Dispatched/Cancelled" mismatch).

## Go-live (cannot be tested on localhost — do at deploy)
1. Set **production** Delhivery credentials + `apiUrl = https://track.delhivery.com` in the prod env (boot fails fast on placeholder/staging-in-prod).
2. Register your **public webhook URL** with Delhivery and confirm **one real tracking callback** updates an order.
3. **E-way bill** — only required when shipping an order **> ₹50,000 interstate** (legal); wire when applicable.
