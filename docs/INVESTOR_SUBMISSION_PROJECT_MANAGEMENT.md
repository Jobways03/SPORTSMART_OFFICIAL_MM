## 1. Roles & RACI Matrix

### 1.1 Core Team Roles

| Role | Abbreviation | Responsibilities |
|------|-------------|------------------|
| **Founder / CEO** | CEO | Final authority on deal terms, company vision narrative, investor meetings, term sheet negotiation |
| **Co-Founder / CTO** | CTO | Technology narrative, product demo, technical due diligence responses, architecture walkthroughs |
| **CFO / Head of Finance** | CFO | Financial model ownership, unit economics, cap table management, legal/compliance coordination |
| **Head of Product** | HProd | Product roadmap, market analysis, competitive positioning, user metrics |
| **Head of Operations** | HOps | Operational metrics, seller/logistics data, supply-chain narrative |
| **Legal Counsel** | Legal | Corporate governance docs, IP assignments, regulatory compliance, NDA/term sheet review |
| **Data Room Manager** | DRM | Data room setup & maintenance, access control, version tracking, document uploads |
| **Investor Relations Lead** | IR | Outreach scheduling, CRM management, follow-up cadence, investor communication |
| **Technical Lead** | TL | Codebase documentation, security audit coordination, scalability evidence |
| **Financial Analyst** | FA | Financial model updates, scenario analysis, market sizing data |

## 5. Version Control & Approval Rules

### 5.1 Versioning Scheme

```
v[Major].[Minor]

Major: Structural changes, significant content updates, new sections
Minor: Corrections, formatting, data refreshes, typo fixes
```

**Examples:**
- `v1.0` - Initial approved version
- `v1.1` - Minor data update after first investor feedback
- `v2.0` - Major restructure after pivot in narrative

### 5.2 Document Lifecycle States

```
DRAFT --> IN REVIEW --> APPROVED --> PUBLISHED --> SUPERSEDED
                |                                      |
                +--- REJECTED (back to DRAFT) ---------+
```

| State | Meaning | Who Can Advance |
|-------|---------|-----------------|
| **DRAFT** | Work in progress, not for external sharing | Author |
| **IN REVIEW** | Submitted for review by stakeholders | Author submits, Reviewer advances |
| **APPROVED** | Reviewed and approved for data room | Accountable person per RACI |
| **PUBLISHED** | Live in data room, visible to investors per access tier | DRM publishes after approval |
| **SUPERSEDED** | Replaced by newer version, retained for audit trail | DRM marks on new version publish |

### 5.3 Approval Authority Matrix

| Document Category | Author | Reviewer(s) | Final Approver |
|-------------------|--------|-------------|----------------|
| Pitch deck & narrative | CEO | CTO, CFO, HProd | CEO |
| Financial model & projections | FA | CFO | CFO |
| Cap table & equity docs | CFO | Legal | CEO |
| Product & technology docs | TL | CTO, HProd | CTO |
| Market & strategy docs | HProd | CEO, FA | CEO |
| Legal & compliance docs | Legal | CFO | Legal + CEO |
| Operational metrics | HOps | HProd, CFO | CEO |
| DD pre-fill responses | Domain owner | Cross-functional peer | Domain CXO |
| Deal documents | Legal | CFO | CEO + Legal |

### 5.4 Change Control Rules

| Rule | Policy |
|------|--------|
| **Single source of truth** | Data room is the only authoritative source. No emailing standalone files to investors. |
| **No retroactive edits** | Once a version is PUBLISHED, it cannot be edited. A new version must be created. |
| **Change log required** | Every version bump must include a change log entry in the document header or tracker. |
| **24-hour review SLA** | Reviewers must provide feedback within 24 hours of submission. Escalate to CEO if blocked. |
| **Dual sign-off for financials** | All financial documents require both FA and CFO approval before publishing. |
| **Legal review for external-facing** | Any document going to investors must have Legal confirmation of no liability risk. |
| **Watermarking** | All Tier 1+ data room documents must be watermarked with investor name and access date. |
| **Download restrictions** | Tier 2+ documents should be view-only where the platform supports it. |
| **Audit trail** | DRM logs every upload, version change, access grant, and download in `00.03_Access_Log.xlsx`. |
| **Expiry review** | All documents with data older than 30 days must be flagged for refresh by DRM. |

### 5.5 Emergency Update Protocol

For time-sensitive updates (e.g., material metric changes, legal disclosures):

1. Author notifies CEO + DRM via designated channel (Slack/email)
2. Fast-track review: 4-hour SLA (not 24-hour)
3. CEO provides verbal approval (documented in writing within 24 hours)
4. DRM publishes immediately with version bump
5. IR notifies affected investors of the update within same business day

## 9. Definition of Done

### 9.1 Deliverable Definition of Done

A deliverable is considered **DONE** when ALL of the following criteria are met:

#### Content Quality

- [ ] Content is accurate, up-to-date, and reflects latest company data (no data older than 30 days)
- [ ] All financial figures are internally consistent across documents (pitch deck matches financial model matches unit economics)
- [ ] No placeholder text, [TBD], or [TODO] markers remain
- [ ] Spelling, grammar, and formatting have been proofread
- [ ] Brand guidelines followed (fonts, colors, logo usage, terminology)
- [ ] No confidential information exposed beyond the intended access tier
- [ ] No competitive or disparaging language that could create legal liability

#### Review & Approval

- [ ] Author self-review completed
- [ ] Peer review completed by designated reviewer(s) per RACI
- [ ] Cross-functional consistency check performed (e.g., CFO confirms financial slides in pitch deck)
- [ ] Final approver has provided written sign-off (email, Slack message, or tracker entry)
- [ ] Legal has confirmed no liability risk for investor-facing documents

#### Data Room Readiness

- [ ] File named per naming convention: `[FolderNumber].[SubNumber]_[Name]_v[X.X].ext`
- [ ] File placed in correct data room folder per Section 4 structure
- [ ] Previous version marked as SUPERSEDED (not deleted)
- [ ] Version number updated in Deliverables Tracker (Section 6.2)
- [ ] Change log entry added documenting what changed
- [ ] Watermarking configured (if Tier 1+ document)
- [ ] Access permissions verified per tier assignment
- [ ] DRM has confirmed upload and accessibility

### 9.2 Milestone Definition of Done

A milestone is considered **DONE** when ALL of the following criteria are met:

- [ ] All deliverables associated with the milestone meet the Deliverable DoD above
- [ ] All action items from the milestone are closed or formally deferred with CEO approval
- [ ] Exit criteria specified in the Workplan (Section 2) are satisfied
- [ ] Milestone status updated in the project tracker
- [ ] Any risks or issues surfaced during the milestone are logged in the Risk Register (Section 6.5)
- [ ] Lessons learned captured (if applicable)
- [ ] Next milestone dependencies are unblocked

### 9.3 Investor Interaction Definition of Done

An investor interaction (meeting, DD session, follow-up) is considered **DONE** when:

- [ ] Meeting minutes completed per template (Section 7) within 4 hours
- [ ] Internal debrief completed within 24 hours
- [ ] External follow-up email sent within 24 hours
- [ ] All requested materials sent within 48 hours (or acknowledged with expected delivery date)
- [ ] Action items logged in Action Item Tracker (Section 6.4) with owners and due dates
- [ ] Investor Pipeline Tracker (Section 6.1) updated with current stage and notes
- [ ] DD Question Tracker (Section 6.3) updated if new questions received
- [ ] Key signals documented (positive indicators, concerns, competitive mentions)
- [ ] Internal team aligned on next steps via debrief

### 9.4 Due Diligence Response Definition of Done

A DD response is considered **DONE** when:

- [ ] Question fully understood (clarification sought from investor if ambiguous)
- [ ] Response drafted by domain owner
- [ ] Response reviewed by at least one cross-functional peer
- [ ] Response reviewed by Legal for liability/sensitivity
- [ ] Supporting documents referenced and linked (data room paths or attachments)
- [ ] Response logged in DD Question Tracker with dates
- [ ] Response sent within 48-hour SLA (or acknowledged with timeline if complex)
- [ ] Investor acknowledgment received (or follow-up sent within 48 hours)

### 9.5 Outreach Definition of Done

An outreach wave is considered **DONE** when:

- [ ] All investors in the wave have been contacted via appropriate channel
- [ ] Investor Pipeline Tracker updated with outreach dates and status
- [ ] Follow-up sequence scheduled for non-responders
- [ ] Response rate tracked and reported in weekly pipeline review
- [ ] Any bounced emails or invalid contacts flagged and replaced
- [ ] Warm intro requests tracked with connector follow-ups scheduled
